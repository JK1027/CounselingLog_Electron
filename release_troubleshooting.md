# 🚀 GitHub Actions 배포 트러블슈팅 및 예방 가이드 (release_troubleshooting.md)

이 문서는 **상담일지 Electron 관리 시스템**의 자동 업데이트 및 배포 파이프라인(GitHub Actions)을 구축하는 과정에서 겪은 빌드 실패 오류의 원인, 시행착오 과정, 그리고 향후 유사 문제를 예방하기 위한 핵심 개발 지침을 정리한 문서입니다.

---

## 1. 발생한 오류와 근본 원인 (Root Cause)

### 🚨 1) 콘솔 한글 인코딩 에러 (`UnicodeEncodeError`) - 치명적
* **증상:** `Build Python Backend` 단계 진입 후 1초도 안 되어 에러를 내며 실패.
* **원인:** 
  * 로컬 개발 컴퓨터는 **한국어 Windows 환경(CP949/UTF-8)**으로 세팅되어 있어 빌드 스크립트(`build_backend.py`) 안의 한글 텍스트(`"빌드 시작"`, `"컴파일 성공"`)를 출력하는 데 문제가 없었습니다.
  * 반면, GitHub Actions의 가상 머신(`windows-latest`)은 **영문 Windows 환경(CP1252)**이 디폴트입니다. 
  * 파이썬 `print()` 함수가 영문 환경의 콘솔에 유니코드 한글 문자를 출력하려 시도하다가, 인코딩이 불가능해 `UnicodeEncodeError: 'charmap' codec can't encode...` 에러를 뿜으며 즉사했던 것입니다.

### 🚨 2) `shell=False`와 시스템 `PATH` 미아 문제
* **증상:** 쉘 이스케이프 문제를 피하기 위해 `shell=False`로 전환한 직후, 파이썬 실행 파일을 찾을 수 없어 0초 만에 종료됨.
* **원인:** 
  * `shell=True`일 때는 cmd.exe가 환경 변수 `PATH`를 자동으로 스캔하여 `"python"` 명령을 찾아서 연결해 줍니다.
  * 하지만 `shell=False`일 때는 OS의 로우레벨 프로세스 생성 API가 직접 기동되므로, 단순 텍스트인 `"python"`만 던지면 경로를 찾지 못해 `FileNotFoundError`가 발생했던 것입니다.
  * 이는 현재 구동 중인 인터프리터의 보장된 절대 경로인 **`sys.executable`**을 사용하도록 고정하여 해결했습니다.

## 2. 시행착오 과정 및 해결 히스토리 (Timeline)

빌드를 성공시키기 위해 총 8차례의 튜닝 작업을 수행하며 점진적으로 환경 의존성을 걷어냈습니다.

1. **Run #1~#2 (절대 경로 및 가상환경 문제):**
   * 로컬의 가상환경 경로(`venv/Scripts/python.exe`)와 아이콘 소스 절대 경로가 하드코딩되어 빌드가 꼬였던 문제를 가상환경 유무에 따른 동적 분기 처리로 1차 해결했습니다.
2. **Run #3~#4 (글로벌 PyInstaller 호출 문제):**
   * 환경 변수 `PATH`에 `pyinstaller` 명령어가 잡히지 않아 실패하던 문제를 `python -m PyInstaller` 모듈 호출 방식으로 우회 구동하였습니다.
3. **Run #5~#6 (쉘 이스케이프 및 실행 경로 파일 누락 문제):**
   * Windows CI 환경의 인자 꼬임을 해결하고자 `shell=False`로 전환하고, `sys.executable`을 도입해 실행 파일 경로 탐색 오류를 완전히 차단했습니다.
4. **Run #7 (디버그 파이프라인 및 복사 우회화):**
   * PyInstaller가 일렉트론 리소스 디렉토리를 직접 건드리다 파일이 잠기는 충돌 문제를 예방하기 위해 임시 빌드 아웃풋(`backend/dist`)을 빌드한 후 `shutil.copytree`로 복사해 오는 간결한 아키텍처로 개편했습니다. 
   * 실패 시 원인을 정확히 볼 수 있도록 GitHub Actions에 **빌드 로그 아티팩트 업로드 스텝**을 추가했습니다.
5. **Run #8 (한글 출력 배제 - 최종 성공 🎉):**
   * 수집된 로그 분석을 통해 인코딩 에러를 규명하고, 빌드 스크립트 내부의 모든 로그 출력을 영문 표준으로 변경해 최종 빌드 성공 및 릴리즈 업로드를 완수했습니다.

## 3. 향후 재발 방지를 위한 개발 가이드라인 (Best Practices)

앞으로 배포 파이프라인을 수정하거나 다른 프로젝트의 빌드 스크립트를 작성할 때 반드시 유의해야 할 **4대 예방 지침**입니다.

### 💡 1. 빌드/배포 스크립트 내 모든 로그 출력은 '영문(ASCII)'으로 통일
* **원칙:** 빌드용 파이썬 스크립트나 배치 파일 내의 `print()` 또는 로그 메시지에는 **한글이나 특수 유니코드 문자를 절대 사용하지 않습니다.**
* **이유:** 빌드가 돌아가는 원격 서버(GitHub Actions, AWS, Azure 등)는 언어 및 콘솔 인코딩 사양이 로컬 컴퓨터와 다를 확률이 매우 높으며, 이는 즉각적인 인코딩 크래시로 연결됩니다.

### 💡 2. 로컬/CI 하이브리드 지원 시, 가상환경 분기는 '인터프리터' 상속으로 해결
* **원칙:** 로컬 venv 가상환경과 글로벌 CI 환경을 분기 처리할 때, 복잡한 경로 탐색 대신 **`sys.executable`**을 부모로 상속받아 모듈 실행 방식으로 구동합니다.
* **예시:**
  ```python
  # 나쁜 예 (가상환경 파일명을 직접 찾음)
  PIP_BIN = "venv/Scripts/pip.exe" if local else "pip"
  
  # 좋은 예 (현재 스크립트를 구동하고 있는 파이썬 엔진에 모듈 탑재 실행 명령 전달)
  pip_cmd = [sys.executable, "-m", "pip", "install", "..."]
  ```

### 💡 3. `shell=False` 환경에서는 반드시 절대 경로와 리스트 인자 사용
* **원칙:** 프로세스 제어의 안정성을 위해 `shell=False`를 사용하되, 실행 파일의 경로는 `sys.executable`이나 `shutil.which` 등을 통한 **절대 경로** 형태여야 하며, 파라미터는 빈칸 단위로 쪼개진 **리스트(`list`)** 형태여야 합니다.

### 💡 4. KISS (Keep It Simple, Stupid) 원칙 준수
* **원칙:** 빌드 스크립트 내부에는 종속성 라이브러리 직접 설치(`pip install`), 설정 파일 동적 수정(`requirements.txt`), 이미지 가공(`pillow`) 등 복잡한 과정을 포함하지 않습니다.
* **역할 분리:**
  * **CI/CD 워크플로우 (`.yml`):** 파이썬/노드 환경 세팅, 라이브러리 설치, 환경 변수 주입 전담.
  * **빌드 스크립트 (`.py`):** 순수한 파일 컴파일(PyInstaller 구동) 및 단순 아웃풋 복사만 전담.
  * **설정 파일 (`.spec`):** hidden-imports, data 리소스 경로, 아이콘 지정 등 정적 메타데이터 관리 전담.
